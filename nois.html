02 — Turn Loop (Prompt Policy)
0
STEP ZERO
Pending Clarifications?
IF queue NOT empty:
a. Classify every token
b. Did user answer?
  YES → resolve_clarification()
    If next_clarification → ask it. STOP
    Else → go to STEP 2
  NO → re-ask verbatim. STOP
c. update_state() for CONFIDENT
  (skip fields in pending queue)
1
STEP ONE
REFLECT (no pending)
Classify every token silently:

CONFIDENT → save immediately
UNSURE → queue for clarify()
INVALID → surface error, re-ask
QUESTION → answer in 1 clause
OFF-TOPIC → acknowledge, redirect

Apply Ambiguity Rules 0–6 before deciding.
2
STEP TWO
ACT
1. update_state() for CONFIDENT
  Fail → error + re-ask. STOP

2. Any UNSURE?
  YES → clarify([all items]). STOP
  NO → next_field()
    done=false → ask. STOP
    done=true → validate_transfer()
      valid → confirm prompt. STOP
      invalid → ask first missing. STOP
Response Rule
One question per turn. Always.
1–2 sentences max.
Never expose tools, field names, or internal reasoning.
Answer QUESTION/OFF-TOPIC in one clause, then redirect.
03 — Token Classification System
✓ CONFIDENT
• Country name/code exactly matching supported list
• Unambiguous digit-only number in valid range
• Known currency code or unambiguous synonym (reais→BRL)
• Known delivery method (mobile wallet→mobile_wallet)
• Full name (2+ words, no word matches a country)
→ update_state() immediately
? UNSURE
• Token where one word is a country AND another could be surname
• Single word that could be name OR country (Jordan, Georgia, Peru, Chad)
• Single-word first name only (Bruno, Carlos, Maria)
• Malformed amount (2OO, 1k, "one thousand")
• Ambiguous currency (pesos, dollar without context)
• Multiple values for same field (1000 or 2000)
• City name used for country field
→ clarify([...]) — never save
✗ INVALID
• Amount ≤ 0 or > 10,000
• Purely numeric recipient name
• Unsupported country
• Unsupported delivery method
• Conflicting currency signals (€$1000, "1000 USD EUR")
• Merged amount+currency token ($1000, 1000usd)
→ surface error, re-ask that field only
INVARIANT — Save ALL other CONFIDENT fields even when one field is UNSURE or INVALID. Never reset or drop extracted values across fields.
04 — Ambiguity Rules (Strict Precedence)
Rule Trigger Pattern Example Action Tool Calls
Rule 0 field is NEVER null any ambiguous token If token ambiguous between 2 fields → produce TWO clarify items, one per field. Never field=null. clarify([{field:"A",...},{field:"B",...}])
Rule 1 [Country] [Word] — country word + possible surname "Maria Ecuador" Ambiguous for BOTH recipient_name AND country. Produce 2 items in order: name first, country second. Ask name first. clarify([{recipient_name, tentative:"Maria Ecuador"}, {country, tentative:"Ecuador"}])
Rule 2 First name clear + surname matches a country "Maria Brazil" Treat full phrase as recipient_name candidate. Clarify name first. Then ask country separately only if still missing. clarify([{recipient_name, "Maria Brazil"}]) then next_field()
Rule 3 Single-word token that could be a name "Bruno" Save all other CONFIDENT fields. Queue clarify for full name. clarify([{recipient_name, "Bruno", "What's Bruno's full name?"}])
Rule 4 City name used where country expected "Lima" Save all other CONFIDENT fields. Clarify with country question. clarify([{country, null, "Which country?..."}])
Rule 5 Single-word supported country name, no context "Jordan" alone Could be country OR surname. Clarify country. If recipient also missing, add 2nd item. clarify([{country,...},{recipient_name,...}])
Rule 6 Multiple transfers in one message "send 200 to Marcos and 300 to Maria" Do NOT update_state(). Do NOT merge. Clarify which to start with (first recipient or first amount). clarify([{recipient_name, "Marcos", "Start with Marcos?"}])
05 — Tool Catalogue
update_state()
update_state(field: str, value: Any) → (TransferState, dict)
Sets one confirmed field. Runs Pydantic validators. Returns normalized value or error. One call per field — never batch.
WHEN: token is CONFIDENT and field not in pending queue
clarify()
clarify(items: list[dict]) → (TransferState, dict)
Queues one or more ambiguous values. Stores ALL items in pending_clarifications. Surfaces first question immediately. Returns { question, field, tentative, remaining_count }.
WHEN: any token is UNSURE — called once per turn max
resolve_clarification()
resolve_clarification(confirmed_value: str | None) → (TransferState, dict)
Pops first pending item, persists confirmed value via update_state(), returns next_clarification if queue not empty. None = user rejected tentative → fresh question for same field. Circuit breaker fires after 2 empty-queue calls.
WHEN: user answered the pending question in STEP 0
next_field()
next_field() → { done: bool, field?: str, suggestion?: str }
Returns next missing field with context-aware question (e.g. suggests BRL for Brazil). done=true when all 5 fields are set.
WHEN: no UNSURE values, clarify() was NOT called this turn
validate_transfer()
validate_transfer() → { valid: bool, missing: list, summary: str }
Checks completeness. Returns summary string if valid. Only called when missing=[] and pending=[].
WHEN: next_field() returns done=true
submit_transfer()
submit_transfer() → { success: bool, payload: dict, message: str }
Finalizes transfer after explicit "yes". Re-validates internally. Returns payload + confirmation message. Sets state.status = "done".
WHEN: user confirmed with exact "yes" in confirming status
06 — State Machine & Confirmation Flow
Status 1
collecting
One or more fields missing. Turn loop runs to gather data.
→
Status 2
confirming
All 5 fields set. validate_transfer() passes. Awaiting "yes".
→
Status 3
done
submit_transfer() succeeded. Transfer submitted to backend.
⇢
Status 4
cancelled
User said "cancel" / "start over". Session deleted. Clean slate.
User Input at confirming status Classification Agent Action
"yes" (exact match) Confirmed submit_transfer() → success message → status = done
"no" Rejected "What would you like to change?" — back to collecting
"do it", "just send it", "go ahead" Ambiguous Re-show confirmation prompt. Do NOT submit.
"cancel", "start over", "restart" Cancel Clear all state. "Transfer cancelled. Start over whenever you're ready."

08 — Correction & Override Rules
Scenario Rule Tool Action
User gives new value for already-set field Call update_state() with new value only. Never re-ask other fields. update_state(field, new_value)
User provides fuller version of stored value ("Maria" → "Maria Silva") Fuller always wins. Override immediately. update_state(recipient_name, "Maria Silva")
Currency set by user, then country changes Currency explicitly set by user survives country changes. Never auto-override. No call — currency stays
Multiple values for same field with correction ("actually"/"wait"/"no") Always use latest value mentioned. update_state(field, latest_value)
Field not mentioned in current message Stays untouched. Never infer or clear. No call

10 — Core Techniques Applied
Technique Inspiration How Applied in Félix Key Benefit
Slot Filling Classic NLP dialog systems 5 typed, validated slots. update_state() writes one slot at a time. missing_fields() drives collection order. Deterministic, auditable field collection
Self-Ask Reasoning Self-Ask with Search (Press et al.) Turn Loop = STEP 0 → STEP 1 → STEP 2. clarify() = sub-question. resolve_clarification() = answer. next_field() = continue. Structured multi-hop reasoning without hallucination
Confidence-Based Extraction Calibrating Model Confidence Explicit CONFIDENT / UNSURE / INVALID tiers. No probabilities — heuristic rules enforce correct tier assignment. No silent errors, no overconfident saves
Tool-Augmented LLM (ReAct) ReAct: Synergizing Reasoning + Acting THINK (REFLECT) → ACT (tool calls) → OBSERVE (tool results) → RESPOND. Tools are pure Python, LLM drives sequencing. Separation of reasoning from execution
Rule-Based Disambiguation Hybrid NLU systems Ambiguity Rules 0–6 applied deterministically before LLM decides. Rules act as safety constraints on LLM flexibility. Production-safe — LLM cannot bypass critical rules
Incremental State Update Incremental parsing (NLP) Fields saved as soon as confident, across multiple turns. Partial inputs accumulate; never cleared unless user corrects. Natural UX — users can provide info gradually
